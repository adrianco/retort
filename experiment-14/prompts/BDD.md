Follow Behaviour-Driven Development (BDD) throughout. Express the required behaviour as Given-When-Then scenarios (Gherkin style): for each requirement, write a scenario that names the precondition (Given), the action (When), and the expected outcome (Then), describing what the system does from an external user's perspective in business-readable language. Then implement the system so every scenario passes. Structure the tests around these behaviour scenarios — GWT-structured PyTest (or a BDD framework such as behave/pytest-bdd in Python, or the idiomatic BDD style of the target language) — one scenario per behaviour, so the test suite reads as a specification of the system's behaviour. For example: "Feature: Match Queries / Scenario: Find matches between two teams / Given the match data is loaded / When I search for matches between two teams / Then I receive a list of matches, and each match has date, scores, and competition." This is the methodology the original brazil-bench template prescribed in-repo; it is captured here for completeness so all four arms (BDD / neutral / TDD / ATDD) are specifiable purely by prompt.
