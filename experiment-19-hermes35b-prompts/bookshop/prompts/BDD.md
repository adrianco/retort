Follow Behaviour-Driven Development (BDD) throughout. Express the required behaviour as Given-When-Then scenarios (Gherkin style): for each requirement, write a scenario that names the precondition (Given), the action (When), and the expected outcome (Then), describing what the system does from an external client's perspective in business-readable language. Then implement the system so every scenario passes. Structure the tests around these behaviour scenarios — GWT-structured PyTest (or a BDD framework such as behave/pytest-bdd in Python, or the idiomatic BDD style of the target language) — one scenario per behaviour, so the test suite reads as a specification of the system's behaviour. For example: "Feature: Book catalogue / Scenario: Add a book / Given the catalogue is empty / When I create a book with a title and author / Then it is stored and returned with a generated id and a 201 status; Scenario: Filter books by author / Given several books by different authors exist / When I list books filtered by one author / Then only that author's books are returned." Exercise the system through its public interface (the REST API's HTTP endpoints and JSON contracts), not its internals, so the scenarios read as a specification of what the service does.
