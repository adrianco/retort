Follow executable Acceptance Test-Driven Development (ATDD) throughout, in the Continuous Delivery style. Before writing any implementation, translate each requirement / acceptance criterion in TASK.md into an automated, executable acceptance test that acts as an executable specification, and let those tests drive the work until they pass. Write the acceptance tests so that they: are written from the perspective of an external user of the system; exercise the System Under Test only through its public interface (here, the MCP tools/protocol) with no back-door access to internals; assert on WHAT the system does, not HOW it does it, using the language of the problem domain (e.g. "find matches between two teams", "get team statistics") rather than implementation mechanics; and are atomic and independent — each scenario starts from a running but empty system and shares no data with other tests. The acceptance tests should fail initially; then implement the system until every acceptance test passes, using finer-grained unit TDD underneath to build the internals. Passing the executable acceptance suite is what demonstrates the requirements are met.
