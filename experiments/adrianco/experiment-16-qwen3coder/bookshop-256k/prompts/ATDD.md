Follow executable Acceptance Test-Driven Development (ATDD) throughout, in the Continuous Delivery style. Before writing any implementation, translate each requirement / acceptance criterion in TASK.md into an automated, executable acceptance test that acts as an executable specification, and let those tests drive the work until they pass. Write the acceptance tests so that they: are written from the perspective of an external client of the service; exercise the System Under Test only through its public interface (here, the REST API — the HTTP endpoints and their JSON request/response contracts and status codes) with no back-door access to internals or the database; assert on WHAT the system does, not HOW it does it, using the language of the problem domain (e.g. "create a book", "list all books", "filter books by author", "update a book", "delete a book", "reject a book with a missing title") rather than implementation mechanics; and are atomic and independent — each scenario starts from a running but empty service and shares no data with other tests. The acceptance tests should fail initially; then implement the service until every acceptance test passes, using finer-grained unit TDD underneath to build the internals. Passing the executable acceptance suite is what demonstrates the requirements are met.
